# Step 6 — Incremental Coding (CODE) [OPTIONAL]

> **Trigger:** `/d2c code`
> 
> **Note:** This step is NOT part of the core D2C workflow. Steps 1-5 produce the full AI context. Code generation is triggered explicitly by the user via `/d2c code`. See the execution roadmap in `.d2c/PLAYBOOK.md` for implementation phases and dependencies.

**Input:** PLAN.md + SPEC.md + AGENTS.md + DESIGN.md + tokens.css + ASSETS.md
**Output:** Incrementally compilable project code

## 6a. Code Generation Process

Follow the PLAN.md task list. Each task is an atomic unit (15-45 min). Show
progress and request confirmation after each task.

### Code generation order

1. **Foundation first:** Design system infrastructure (atomic components from
   SPEC.md component tree → animated components)
2. **Layout & routing:** Layout components → page skeletons → navigation
3. **Pages & features:** Per page, feature components → hooks → data fetching
4. **Integration:** API endpoints (if applicable) → state management → tests
5. **Polish:** Responsive adjustments → edge cases → final animation integration

## 6b. Pre-Coding Setup

Before writing the first line of code:

1. **Confirm token files exist** (`styles/tokens.css` and/or `styles/tokens.ts`
   from Step 2). If they don't, stop and regenerate from Step 2 — **no code
   should be written without tokens in place.**
2. **Re-read token files** at the start of every session to keep their values
   in context.
3. **Confirm Tailwind config** (if applicable) extends from `tokens.css`
   values, not hard-coded numbers.

## 6c. Token Adoption Rules (MANDATORY)

Every component file generated in this step MUST follow the Token Adoption rules
defined in `guides/STEP_4_SPEC.md` section 4d. Summary of hard rules:

| # | Rule | Check |
|---|------|-------|
| 1 | **Zero hex/rgba literals** in component styles | `grep -nP '#[0-9a-fA-F]{3,6}|rgba?\(|rgb\(` per file |
| 2 | **Zero px/rem literals** for spacing, font-size, radius | Check style properties use `var(--spacing-*)`, `var(--font-size-*)`, `var(--radius-*)` or TS token imports |
| 3 | **Typography must use tokens** | font-family, font-weight, line-height must reference tokens |
| 4 | **Shadows must use tokens** | box-shadow must reference `var(--shadow-*)` or `tokens.shadows.*` |
| 5 | **Token files are READ-ONLY** | Never modify `styles/tokens.css` or `styles/tokens.ts` during coding |

## 6d. Token Fidelity Check (Post-Generation Gate)

After each component file is generated AND before marking it as complete, run
this automated check:

```
Step 1: Read the generated file.
Step 2: Scan for literal violations:
  - Patterns: #[0-9a-fA-F]{3,8}, rgb(, rgba(, hsl(, hsla(
  - px/rem literals in style/className properties
  - Any color/spacing/size value that could be a token
Step 3: For each violation found:
  - If token equivalent exists in tokens.css/tokens.ts → REPLACE WITH TOKEN REFERENCE
  - If NOT an exception (see 4d) → FIX BEFORE PROCEEDING
  - If an exception → ADD CODE COMMENT explaining why
Step 4: Only after zero violations → mark task complete
```

**If a violation is found in a component you just wrote, fix it immediately.**
Do not defer.

Example fix workflow:

```
# ❌ DETECTED VIOLATION
❌ src/components/ui/Button.tsx: background: #6366f1

# ✅ FIXED
✅ src/components/ui/Button.tsx: background: var(--color-primary)

# Confirm user, then proceed to next PLAN.md task
```

For existing project files (not generated by D2C), flag violations as warnings
in the output but do not modify the files unless the user explicitly confirms.

## 6e. Explicit Request for 3rd-Party Component Overrides

When integrating third-party UI libraries, their predefined styles often bypass
token variables. For each such component:

1. Check if the library exposes a `theme` provider or customization API.
2. Configure it to use token variables where possible.
3. If hard-coded overrides are unavoidable (e.g., overriding a library's
   internal CSS class), wrap them in a separate `.overrides.css` file scoped
   to the library, with a code comment referencing the design token.

```
/* src/styles/overrides.css — library component overrides */
/* Override required because react-select@5 uses hard-coded colors */
.react-select__control {
  background: var(--color-white);
  border-color: var(--color-neutral-300);
}
```

## 6f. Quality Checks

After each task compiles successfully:

- **Token fidelity:** Confirm no new hard-coded values were introduced
- **Build verification:** `npm run build` or framework equivalent must pass
- **Component isolation:** The new component renders without breaking existing UI
- **TypeScript:** `npx tsc --noEmit` (if TS project)

## 6g. Existing Project Integration

When coding into an existing project:

- Match existing file naming, import patterns, and directory layout
- Import and extend the existing `tokens.css` (it was added in Step 2)
- If an existing component already exists for the same purpose, reuse it and
  apply token-driven styling rather than rewriting
- Append to existing files rather than creating parallel structures
